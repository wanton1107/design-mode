# Go语言设计模式

## 单例模式(singleton)

- 懒汉模式
- 饿汉模式
- 双检查模式
- sync.Once

## 简单工厂模式(simple factory)

简单工厂不是一种设计模式，反而像是一种编程习惯。
简单工厂模式又叫静态工厂方法模式，通过使用静态方法接收不同的参数来返回不同的实例对象。

### 优点

封装了对象创建过程，可以通过参数直接获取对象。把对象的创建和业务逻辑分开，这样以后就避免了修改客户代码，如果要实现新产品直接修改工厂类，而不需要在源代码中修改

### 缺点

增加新产品时还是需要修改工厂类的代码，违背了“开闭原则”

## 工厂方法模式(method factory)

定义一个用于创建对象的接口，让子类决定实例化哪个产品类对象。工厂方法使一个产品类实例化延迟到其工厂的子类。每次增加新产品需要增加新的工厂类型。

### 优点

- 只需要知道具体工厂的名称就可得到所要的产品，无需知道产品的具体创建过程
- 在系统增加新的产品时只需要添加具体产品类和对应的具体工厂类，无须对原工厂进行任何修改，满足开闭原则

### 缺点

- 每增加一个产品就要增加一个具体产品类和一个对应的具体工厂类，增加了系统复杂度

## 抽象工厂模式

对工厂方法模式的扩展，相当于一个超级工厂，用于创建其他工厂的模式
提供统一的接口创建一系列相关的产品，比如多个操作系统平台需要创建一系列主题不同的组件，或者游戏中服装套装的替换，通常一整套一起换

### 优点

- 对于不同产品系列有较多共性时，比如不同平台的按钮组件，可以使用抽象工厂模式，有助于提升组件的复用性
- 当需要提升代码的扩展性并降低维护成本时，把对象的创建和使用过程分开，能有效地将代码统一到一个级别上
- 解决跨平台带来的兼容性问题

### 缺点

- 增加新的产品等级结构麻烦，需要对原有结构进行较大的修改，甚至需要修改抽象层代码，违背了开闭原则

## 生成器模式

逐步构建复杂的对象

### 优点

- 分步创建，更加灵活
- 可以复用相同的制作代码
- 严格遵循单一职责原则

### 缺点

- 代码整体复杂度增加

### 为什么不直接使用setter

- 破坏不可变性：对象在创建后仍可随意修改，容易引发线程安全问题。例如在分布式环境下，User 对象被多处共享，某处调用 setEmail() 会意外修改其他线程看到的值

- 对象状态可能不完整：setter 无法强制要求调用方设置必填字段。比如忘记调用 setUsername() 就直接使用 user 对象，程序可能运行时才出错

- 校验逻辑分散：每个 setter 内部只能校验单个字段，而涉及字段间关联的校验（如“年龄>18时必须填写手机号”）很难实现。Builder 的 build() 方法可以集中进行跨字段校验

- 无法表达构建步骤顺序：某些对象需要按特定顺序设置属性（如先设缓存后设缓存策略），setter 无法控制调用顺序，而 Builder 可以将顺序逻辑写在 build() 或指挥者类中

- 代码可读性差：当字段很多时，setter 方式的创建代码变得冗长且不易辨识哪个字段对应哪个值

## 原型模式

能够复制对象，但是不依赖对象所属的类

### 优点

- 节省对象创建时的成本，比如有很多属性要逐个赋值
- 增加系统的灵活的

### 缺点

- 增加系统的复杂度，需要实现Clone

## 对象池模式

对象被预先初始化并存储于对象池中，当需要时，客户端可以从对象池中请求一个对象并使用，减少频繁的创建及销毁对象造成的资源浪费

### 优点

- 有助于提高应用程序的整体性能
- 有助于限制对象的最大创建数量

### 缺点

- 会增加分配/释放对象的资源开销
- 对象池中的对象数量是有限的，可能会成为一个可伸缩性能瓶颈
- 对象池数量比较难评估，如果过少不能充分发挥对象池的作用；过多则会占用大量的内存资源

## 组合模式

递归组合、树状结构、统一处理

### 使用场景

- 需要忽略组合对象和单个对象之间的差异，以相同的方式使用多个对象，并且用几乎相同的代码处理每个对象
- 需要实现树形结构，只需要调用树顶层对象的方法即可对整棵树进行统一操作

### 优点

- 无须了解构成树形结构的对象的具体类，也无须了解对象是简单的文件，还是复杂的文件，只需调用通用接口中的方法，并且以相同的方式对其进行处理

## 适配器模式

通过适配器类使原本因接口不兼容而不能一起工作的类可以一起工作，比如一个系统只能输出XML格式，另一个系统只能处理JSON格式，如果需要两个系统协同工作，可以使用适配器模式

### 使用场景

- 当开发者希望使用某个类，但是其接口与其他代码不兼容时，或者当开发者使用两个不兼容的系统、类或接口
- 当系统需要使用一些现有的类，而这些类的接口不符合系统的要求，甚至没有这些类的源代码时

### 两种方式

对象适配器：通过组合实现
类适配器：通过继承实现

### 相关角色

- 请求者Client：调用服务的角色
- 目标Target：定义了Client要使用的功能
- 转换对象Adaptee：需要被适配器转换的对象
- 适配器Adapter：实现转换功能的对象

## 桥接模式

通过组合的方式对类进行扩展，抽象和实现分离，通过一个接口连接两套实现，并且可以分别变化

### 使用场景

- 一个系统需要在两个（或更多）独立变化的维度上进行扩展，避免子类爆炸
- 比如不同的遥控设备控制不同的设备，其中遥控代表抽象层，设备代表实现层

### 相关角色

分别代表两个业务

- 抽象层
- 实现层

### 优点

- 提高代码可伸缩性，在添加功能时无需担心破坏程序的其他部分
- 当实体的数量基于两个概念的组合（如形状和颜色）时，桥接可以减少子类的数量
- 桥接可以分别处理两个独立的层次结构--抽象和实现
- 桥接模式可以降低类之间的耦合度--两个类耦合的唯一地方是桥

### 缺点

- 可以能会对程序的性能产生负担
- 由于需要在两个类之间切换，因此会降低代码可读性

## 装饰器模式

### 使用场景

- 如果开发者希望在不修改代码的情况下使用对象，并且在运行时为对象添加额外的功能

### 相关角色

- 组件：定义一个接口，用于规范准备接收附加责任的对象
- 具体组件：实现组件接口的类，通过装饰器为其添加一些职责
- 装饰器：是一个包含具体组件对象的类，可以通过其子类扩展具体组件的功能
- 具体装饰器：实现装饰器类的相关方法，并且给具体组件对象添加附加的职责

### 优点

- 为扩展功能提供了一种灵活的替代子类的方法
- 允许在运行时修改方法，而不是返回现有代码并进行修改
- 可以通过装饰动态给类增加功能

### 缺点

- 增加系统复杂度，会创建许多小对象
